C판정 — 퍼지 AND는 곱셈이다

"전부 굽었나?" 같은 조건문이 없어도 됩니다. 0~1 값들의 AND는 곱셈이니까:

바위 = (1-검지) × (1-중지) × (1-약지) × (1-새끼) × active
보   =   검지  ×   중지  ×   약지  ×   새끼  × active
가위 =   검지  ×   중지  × (1-약지) × (1-새끼) × active

전부 1에 가까울 때만 곱이 1에 가깝다 = AND
(1-x)는 NOT. active를 곱하면 손 없을 때 전부 0 = 게이트 공짜
노드구성
inv_fingersMath Range 0~1 → 1~0 = 뒤집기(NOT). null_fingers에 연결
merge_rock → mul_rockMerge(inv_fingers 4ch + sel_active 1ch) → Math Combine Channels = Multiply → rename rock
merge_paper → mul_paperMerge(null_fingers + active) → Multiply → paper
merge_scissors → mul_scissorsMerge(검지·중지는 null_fingers에서, 약지·새끼는 inv_fingers에서 Select + active) → Multiply → scissors

셋을 Merge → null_rps. 판정 텍스트는 06·07과 같은 Text TOP — 문턱만 0.4로 (곱셈 점수는 확률보다 겸손하게 나오니까).

이 패턴 어디서 봤지? 05_Pinch의 Logic(스위치)이 하던 일을 곱셈이 부드럽게 대신합니다. Logic은 딱딱한 0/1, 곱은 연속적인 확신 — 미터가 부드럽게 반응하는 이유.

E안 될 때

  • 중간값을 봐라 — 이 버전의 특권 — len_tip_index(거리), ratio_index(비율), null_fingers(펴짐), null_rps(판정) 뷰어를 차례로 열면 어디서 죽었는지 바로 보입니다. Script 버전은 이게 안 됨.
  • Subtract가 이상해요 — 2입력 Math의 Match by가 Index인지 확인 (채널명이 달라 Name 매칭은 실패).
  • 비율이 0이에요 — 손이 없으면 좌표가 전부 0 → 거리 0. 정상입니다 (active 게이트가 판정을 막아줌).
  • 바위만 계속 떠요 — math_norm From Range 1(0.95)을 올려보세요. 손가락이 "덜 굽음"으로 새는 중.
